## Data Flow Diagrams

### A. Opening a File
```
User clicks file in tree
    ↓
FileTree component emits 'file:open' event
    ↓
Check if tab already exists for this file
    ↓
    YES → Switch to existing tab
    NO  → Create new MarkdownTab
          ↓
          IPC call: 'file:read' → Main Process
          ↓
          Main reads file content → Returns to Renderer
          ↓
          Update tab.content in state
          ↓
          MarkdownEditor renders content
```

### B. Terminal Input/Output
```
User types in terminal
    ↓
XtermComponent captures input
    ↓
IPC call: 'terminal:write' + ptyId + data → Main Process
    ↓
Main writes to pty process
    ↓
Shell processes command
    ↓
Output from pty → Main captures via 'data' event
    ↓
IPC emit: 'terminal:data' + ptyId + output → Renderer
    ↓
Find terminal tab by ptyId
    ↓
XtermComponent writes output to display
```

### C. Saving Markdown
```
User hits Cmd+S or clicks Save
    ↓
Get active tab content
    ↓
If no filePath → Show save dialog (IPC)
    ↓
IPC call: 'file:write' + path + content → Main Process
    ↓
Main writes file to disk
    ↓
Update tab.isDirty = false
    ↓
Update tab title (remove * indicator)
```

## Key Architectural Decisions

### 1. Process Separation
```
MAIN PROCESS (Node.js privileges)
├── File system access
├── Terminal process management (node-pty)
├── Window lifecycle
└── Native OS integration

RENDERER PROCESS (Sandboxed browser)
├── UI rendering (React)
├── User input handling
├── Editor state management
└── Terminal display (xterm.js)

Communication: IPC (Inter-Process Communication)
```

### 2. Tab State Management
```
Centralized State Pattern:

                    ┌─────────────────┐
                    │   TabManager    │
                    │  (React Context)│
                    └─────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
  ┌─────────┐         ┌─────────┐         ┌─────────┐
  │  Tab 1  │         │  Tab 2  │         │  Tab 3  │
  │Markdown │         │Terminal │         │Terminal │
  └─────────┘         └─────────┘         └─────────┘

Operations:
- createTab(type, metadata)
- closeTab(tabId)
- switchTab(tabId)
- updateTab(tabId, changes)
- getAllTabs()
```

### 3. File Browser State
```
Lazy Loading Strategy:

Initial Load
    ↓
Load only root directory children
    ↓
User clicks folder
    ↓
Expand folder: Load its children on-demand
    ↓
Cache in FileNode tree
    ↓
Watch for file changes (fs.watch in Main)
    ↓
Emit updates to Renderer
    ↓
Update tree without full reload
```

### 4. Terminal Lifecycle
```
Terminal Creation Flow:

TerminalTab created
    ↓
Assign unique ptyId
    ↓
Main Process: pty.spawn(shell, {cwd})
    ↓
Store in terminals Map: ptyId → pty instance
    ↓
Setup event handlers:
  - pty.on('data') → Send to Renderer
  - pty.on('exit') → Mark tab as dead
    ↓
Renderer: XtermComponent attaches
    ↓
Bidirectional data flow established

Terminal Cleanup:
Tab closed
    ↓
IPC: 'terminal:close' + ptyId
    ↓
Main: pty.kill()
    ↓
Remove from terminals Map
    ↓
Clean up event listeners
```
