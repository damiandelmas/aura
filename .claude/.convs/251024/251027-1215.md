● Open Design Questions (Systems View)

  1. Result Chaining Model

  Question: How does Operation B reference Operation A's output metadata?

  Options:
  - Implicit context: Last operation's result automatically available to next operation
  - Pipe model: Data flows between operations via stream
  - Explicit storage: Operations write to named locations, subsequent operations read by
  name

  Trade-off: Implicit = simple, brittle. Explicit = flexible, verbose. Pipe = composable,
  requires streaming.

  ---
  2. Graph Identity Strategy

  Question: Who assigns identifiers to constructed graphs?

  Options:
  - System-generated: Random hash, collision-free, no user thought
  - User-specified: Meaningful names, rememberable, requires naming decisions
  - Hybrid: Default to auto, allow override

  Trade-off: Auto = no cognitive load, harder to reference later. User = memorable,
  requires forethought. Hybrid = flexibility tax.

  ---
  3. Result Metadata Access Pattern

  Question: When Operation B needs Operation A's metadata (file path, session ID,
  timestamp), what's the mechanism?

  Options:
  - Variable binding: Operation A binds results to named variables, Operation B references
  variables
  - Positional reference: Operation B implicitly sees "previous result"
  - Explicit pass-through: Operation A outputs ID, Operation B queries by ID

  Trade-off: Variables = flexible but stateful. Positional = simple but inflexible.
  Pass-through = explicit but verbose.

  ---
  4. Parallel Query Execution (NEW)

  Question: How do we execute multiple independent queries simultaneously?

  Context:
  - Single query: search "JWT" → filter by file → filter by session (sequential dependency)
  - Parallel queries: search "JWT" + search "auth" + search "security" (no dependency)

  Requirement: Enable Claude Code to spawn multiple search operations concurrently, gather
  results, then process.

  Architectural pattern:
  - Bash supports background processes: command1 & command2 & wait
  - Fork multiple queries, wait for all completion
  - Aggregate results for subsequent operations

  Use case:
  - "Show me all decisions about authentication, rate limiting, and security"
  - Three parallel searches, merge results, apply graph algorithms

  Constraint: Only read operations parallel. Write operations sequential.

  Parallelism: Built into primitives or bash-level composition (& operator)?

● Design Decisions (Architectural View)

  1. Data Flow Model

  Implicit (stateful):
  - Operations share session state
  - Last result automatically available
  - Simpler syntax, hidden coupling

  Explicit (stateless):
  - Each operation self-contained
  - Results passed explicitly
  - Verbose, clear dependencies

  Choice affects: Composition complexity, debugging clarity, reusability

  ---
  2. Graph Naming

  Auto-hash:
  - System generates IDs
  - No naming decisions
  - Ephemeral by default

  User-named:
  - Meaningful identifiers
  - Requires naming
  - Persistent by intent

  Choice affects: Graph lifecycle, discoverability, reuse patterns

  ---
  3. Metadata Binding

  How Operation N accesses Operation N-1's metadata (session_id, file_path)?

  Scope of coupling: Tight (implicit) vs loose (explicit identifiers)

  Choice affects: Pipeline flexibility, composition reusability

  ---
  4. Parallel Execution

  Bash-level (& operator):
  - External to primitives
  - Claude composes parallelism
  - Observable in usage.log

  Primitive-level:
  - Built into commands
  - System manages concurrency
  - Hidden from observation

  Choice affects: Composability, observability, learning substrate

  ---
  Recommendation Pattern

  For AI-directed systems:
  - Explicit > implicit (AI agents benefit from clarity)
  - Stateless > stateful (compositions more portable)
  - Bash-level parallelism (observable, composable)
  - Hybrid naming (auto with optional override)

  Rationale: AI agents excel at explicit orchestration. Hide less, expose more primitives.

  Direct decision required on each.
