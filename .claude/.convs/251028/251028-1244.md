  ## Summary

  Finalized AURA soft-graph architecture with focus on query-adaptive chunk
   bundling that transcends traditional knowledge graphs. Key insight:
  Traditional KGs serve chunks with fixed context (pre-computed edges),
  AURA serves chunks with dynamic context (runtime graph construction
  adapted to query intent).

  **Architectural innovation**: Query-adaptive bundling strategies—same
  chunks, different context assemblies based on query type (authority,
  explanation, bridge discovery, timeline).

  **Validation**: Real usage (251027-1246) showed 13+ occurrences of
  context assembly needs, validating siblings primitive and graph
  operations.

  **Implementation**: 4 CLI primitives (~450 lines, 3.5 days) enable
  infinite bundling compositions.

  ## Key Decisions

  ### Query-Adaptive Bundling Over Fixed Graph Schemas

  **Context**: Traditional knowledge graphs (Neo4j, Microsoft GraphRAG)
  pre-compute edges at index time, serve chunks with fixed relationship
  bundling.

  **Decision**: AURA constructs graph structure at query time, adapting
  bundling strategy to query intent.

  **Rationale**:
  - Knowledge graphs lock you into one schema (edges defined upfront)
  - Different queries need different context assemblies
  - Runtime graphs enable query-specific bundling strategies
  - Zero precomputation overhead (O(0) vs O(n²))
  - Infinite flexibility (any metadata dimension = potential edge type)

  **Alternatives considered**:
  - Pre-compute knowledge graph → Rejected (maintenance burden, rigid
  structure)
  - Hybrid (pre-compute + runtime) → Rejected (complexity, both overheads)
  - No graph operations → Rejected (loses authority ranking, bridge
  discovery)

  **Trade-offs**:
  - Cost: O(k²) graph construction per query (k = top results, typically
  20-50)
  - Benefit: Infinite bundling strategies, zero maintenance, adapts to
  query context

  **Examples of adaptive bundling**:

  1. **Authority query**: PageRank bundling
     ```bash
     # Query: "What's the most authoritative auth pattern?"
     imem batch '{
       "queries": [
         {"text": "auth", "filters": {"decisions": true}},
         {"text": "auth", "filters": {"failures": true}},
         {"text": "auth", "filters": {"patterns": true}}
       ],
       "combine": true,
       "graph": {"algorithm": "pagerank", "top": 3}
     }'
     # Then: imem siblings <top-result-id>

     Returns: Most-referenced decision + constraints + patterns from same
  file

     Knowledge graph can't do this: Edges are fixed, can't dynamically rank
   by reference count

  2. Explanation query: Context-complete bundling
  # Query: "Explain JWT implementation completely"
  decision=$(imem develop search "JWT" --decisions --limit 1)
  siblings=$(imem siblings <result-id>)
  origin=$(imem filter --session <session-id> --source conversation)

  2. Returns: Decision + implementation + constraints + origin conversation

  2. Knowledge graph limitation: Would need to predefine all these edge
  types
  3. Bridge query: Centrality bundling
  # Query: "Find connecting concepts between chunking and indexing"
  imem batch '{
    "queries": [
      {"text": "chunking section parsing", "limit": 10},
      {"text": "indexing batch processing", "limit": 10}
    ],
    "combine": true,
    "graph": {"algorithm": "centrality", "top": 5}
  }'

  3. Returns: Concepts with highest betweenness centrality (bridges between
   topics)

  3. Knowledge graph can't do this: Would need to precompute centrality for
   all topic pairs
  4. Timeline query: Temporal bundling
  # Query: "Show evolution of chunking decisions"
  decision=$(imem develop search "section chunking" --decisions --limit 1)
  evolution=$(imem filter --session <session-id> --timestamp-after <ts>
  --semantic-similar <result-id>)

  4. Returns: Original decision + later refinements (temporal + semantic
  chain)

  4. Knowledge graph limitation: Static edges can't adapt to "show me what
  came after"

  The paradigm shift:
  - Traditional KG: Static graph → queries adapt to structure
  - AURA: Dynamic graph ← structure adapts to query

  Batch Primitive as Parallelization Infrastructure

  Context: Initially misunderstood batch as multi-query wrapper. Actual
  purpose: generic parallelization of ANY CLI operations.

  Decision: Batch executes any primitives in parallel via single JSON call.

  Rationale:
  - Reduces latency: 3 ops @ 100ms = 110ms parallel vs 300ms sequential
  - Single bash invocation (efficient for Claude Code)
  - Works with any future primitives (not locked to multi-query)
  - Structured JSON I/O (easily parseable)

  Implementation:
  # "queries" format (sugar for common case: multi-search + combine + rank)
  imem batch '{
    "queries": [
      {"text": "auth security", "filters": {"decisions": true}},
      {"text": "auth impl", "filters": {"layer": "implementation"}}
    ],
    "combine": true,
    "graph": {"algorithm": "pagerank", "top": 10}
  }'

  Why "queries" not "parallel":
  - 90% of batch use cases: multi-query + rank
  - "queries" is concise sugar, no "op" boilerplate
  - Generic "parallel" format adds verbosity without benefit

  Trade-offs:
  - Can't parallelize mixed operation types in v1 (search + siblings +
  filter)
  - Mitigation: Add "parallel" format in v2 if validated need emerges

  Template Enforcement Enables Soft-Graph

  Context: Post-hoc systems (MindsDB, Azure) extract metadata
  probabilistically after document creation.

  Decision: Enforce template compliance at creation (document ingestion).

  Rationale:
  - Required fields GUARANTEED present (Context, Solution for Decisions)
  - Optional fields TRACKED (has_alternatives, has_rationale)
  - Enables deterministic filtering (metadata = reliable edge discovery)
  - Powers soft-graph (relationships via metadata queries, not
  probabilistic extraction)

  Schema example:
  Decisions:
    Required: Context, Solution
    Optional: Rationale, Alternatives, Benefits, Drawbacks

  Constraints:
    Required: Description, Impact
    Optional: Mitigation, Workaround

  This is unique: No other system enforces schema at creation → no other
  system has deterministic metadata → no other system can do soft-graph
  efficiently.

  Trade-offs:
  - Domain-specific (coding workflows only, not general documents)
  - Lower flexibility (must follow template)
  - Benefit: Deterministic queries, reliable relationships, unique
  positioning

  Primitive 2: Graph Build (Runtime Graph Construction)

  Purpose: Construct NetworkX graph from search results for algorithm
  application.

  Validation: Need for authority ranking (most-referenced decisions) and
  bridge discovery (connecting concepts).

  Implementation:
  def build_graph(result_ids):
      """Construct temporary graph from query results"""
      import networkx as nx

      results = [load_result(rid) for rid in result_ids]
      G = nx.DiGraph()

      # Add nodes
      for i, r in enumerate(results):
          G.add_node(i,
              result_id=r.id,
              score=r.score,
              file_path=r.payload['file_path'],
              session_id=r.payload.get('session_id'),
              timestamp=r.payload.get('timestamp')
          )

      # Materialize edges from metadata relationships
      for i, r1 in enumerate(results):
          for j, r2 in enumerate(results):
              if i == j:
                  continue

              # FILE edge: same document
              if r1.payload['file_path'] == r2.payload['file_path']:
                  G.add_edge(i, j, type='file', weight=0.8)

              # SESSION edge: same conversation
              if r1.payload.get('session_id') ==
  r2.payload.get('session_id'):
                  G.add_edge(i, j, type='session', weight=0.9)

              # TEMPORAL edge: evolution (later + semantic similar)
              if r2.payload.get('timestamp', 0) >
  r1.payload.get('timestamp', 0):
                  # Would check semantic similarity > 0.7
                  G.add_edge(i, j, type='temporal', weight=0.7)

      # Store with embedded metadata
      G.graph['created'] = datetime.now().isoformat()
      G.graph['node_count'] = len(results)

      graph_id =
  hashlib.md5(str(sorted(result_ids)).encode()).hexdigest()[:8]
      nx.write_gpickle(G, f"~/.context/imem_graphs/{graph_id}.pkl")

      return graph_id

  CLI:
  imem graph build <result-id-1> <result-id-2> ... <result-id-n>
  # Returns: graph-id (e.g., "a7f3d2c1")

  Complexity: O(k²) where k = number of results (typically 20-50)
  - 20 nodes → 400 potential edges → ~10ms construction
  - Compare: Traditional KG with 10,000 docs → 100M edges → hours
  precomputation

  Primitive 3: Graph Apply (Algorithm-Based Ranking)

  Purpose: Apply NetworkX algorithms to constructed graph, return ranked
  results.

  Algorithms:
  - PageRank: Authority (most-referenced nodes)
  - Betweenness Centrality: Bridge concepts (nodes connecting disparate
  topics)
  - Community Detection: Clusters (tightly connected subgraphs)

  Implementation:
  def apply_algorithm(graph_id, algorithm):
      """Apply algorithm, return ranked result IDs"""
      import networkx as nx

      G = nx.read_gpickle(f"~/.context/imem_graphs/{graph_id}.pkl")

      if algorithm == 'pagerank':
          scores = nx.pagerank(G, weight='weight')
      elif algorithm == 'centrality':
          scores = nx.betweenness_centrality(G, weight='weight')
      elif algorithm == 'communities':
          communities = nx.community.louvain_communities(G.to_undirected())
          # Return results grouped by community
          return group_by_community(G, communities)
      else:
          raise ValueError(f"Unknown algorithm: {algorithm}")

      # Return result IDs sorted by score
      ranked_nodes = sorted(scores.items(), key=lambda x: x[1],
  reverse=True)
      result_ids = [G.nodes[node_id]['result_id'] for node_id, score in
  ranked_nodes]

      return result_ids

  CLI:
  imem graph apply <graph-id> pagerank
  imem graph apply <graph-id> centrality
  imem graph apply <graph-id> communities
  # Returns: Ranked list of result IDs

  Performance:
  - PageRank on 20-node graph: ~5ms
  - Centrality on 20-node graph: ~8ms

  Primitive 4: Batch (Parallelization Infrastructure)

  Purpose: Execute multiple operations in parallel via single bash call.

  Implementation:
  def batch_execute(config_json):
      """Execute queries in parallel, optionally combine + rank"""
      config = json.loads(config_json)

      # Parallel execution (async internally)
      results = []
      for query_spec in config['queries']:
          result = search(
              query_spec['text'],
              filters=query_spec.get('filters', {}),
              limit=query_spec.get('limit', 10)
          )
          results.append(result)

      # Combine if requested
      if config.get('combine'):
          merged = combine_results(results)
      else:
          return {'results': results, 'count': len(results)}

      # Graph operations if requested
      if config.get('graph'):
          result_ids = [r.id for r in merged]
          graph_id = build_graph(result_ids)
          ranked_ids = apply_algorithm(
              graph_id,
              config['graph']['algorithm']
          )
          top_k = config['graph'].get('top', 10)
          return [load_result(rid) for rid in ranked_ids[:top_k]]

      return merged

  CLI:
  imem batch '{
    "queries": [
      {"text": "auth security", "filters": {"decisions": true}},
      {"text": "auth impl", "filters": {"layer": "implementation"}}
    ],
    "combine": true,
    "graph": {"algorithm": "pagerank", "top": 10}
  }'

  Performance:
  - 3 queries @ 100ms each
  - Sequential: 300ms
  - Parallel: 110ms (100ms + 10ms overhead)

  Storage Strategy

  Graph persistence:
  ~/.context/imem_graphs/
  ├── <graph-id>.pkl  # NetworkX pickle
  └── usage.log       # Operation log

  Graph metadata (embedded in NetworkX object):
  G.graph['created'] = timestamp
  G.graph['query'] = original query context
  G.graph['node_count'] = number of results

  Lifecycle:
  - Created: imem graph build → persist to disk
  - Used: imem graph apply (can apply multiple algorithms to same graph)
  - Cleaned: Manual imem graph gc or automatic (30+ days unused)

  Why persist: Session continuity (build once, apply different algorithms),
   observability (track graph patterns), pattern mining (analyze structures
   over time).

  Not precomputed: Graphs are query-specific, ephemeral by design, not
  corpus-wide.

  Constraints

  Graph State Ephemeral by Design

  Limitation: Graphs constructed per query, not maintained as persistent
  corpus-wide structure.

  Rationale: Avoids precomputation overhead, adapts to query context, zero
  maintenance.

  Impact: Can't query "the knowledge graph" directly (like Neo4j Cypher).
  Must construct graph from search results first.

  Example workflow:
  # Step 1: Search (get candidates)
  results=$(imem develop search "auth" --decisions)

  # Step 2: Build graph (from candidates)
  graph=$(imem graph build $results)

  # Step 3: Apply algorithm (rank)
  ranked=$(imem graph apply $graph pagerank)

  Mitigation: Slash commands abstract 3-step workflow (user sees single
  command).

  Patterns

  Soft-Graph via Metadata Queries

  Pattern: Relationships discovered dynamically via metadata filters, not
  precomputed edges.

  Edge types = metadata predicates:
  # Traditional KG: Precomputed edge
  decision --[HAS_CONSTRAINT]--> constraint

  # AURA: Metadata query materializes relationship
  constraints = search(filters={
      'file_path': decision.file_path,
      'section_type': 'Constraints'
  })

  Edge catalog:
  - FILE: file_path == X
  - SESSION: session_id == Y
  - TEMPORAL: timestamp > Z AND semantic_similarity > 0.7
  - SEMANTIC: cosine_similarity > threshold

  Benefits: Zero storage, zero maintenance, infinite flexibility (any
  metadata dimension = potential edge).

  Reusable insight: When metadata is rich and deterministic
  (template-enforced), precomputed graphs are unnecessary—query-time
  discovery is sufficient and more flexible.

  Query-Adaptive Bundling Strategies

  Pattern: Same chunks, different context assemblies based on query intent.

  Bundling strategies:

  | Query Type  | Algorithm  | Bundling Strategy             | Use Case
                          |
  |-------------|------------|-------------------------------|-------------
  ------------------------|
  | Authority   | PageRank   | Most-referenced + siblings    | "What's the
  authoritative pattern?" |
  | Explanation | Siblings   | Decision + impl + constraints | "Explain
  this completely"           |
  | Bridge      | Centrality | Connecting concepts           | "How do X
  and Y relate?"            |
  | Timeline    | Temporal   | Evolution chain               | "Show
  decision progression"         |

  Implementation: Same primitives (search, siblings, graph), different
  compositions.

  Reusable insight: Static graph schemas force queries to adapt to
  structure. Dynamic graphs adapt structure to query intent. The latter is
  more powerful for knowledge retrieval.
