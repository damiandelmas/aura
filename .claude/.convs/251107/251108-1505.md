  ---
  (1) Storage Primitive: JSONL

  Why not SQLite/SQL:

  Your corpus is already append-only logs:
  - Changelogs (JSONL-like with frontmatter)
  - Conversations (JSONL)
  - Git commits (append-only DAG)
  - Access logs (append-only)

  Match the primitive to the data:

  .nexus/
  ├── access.jsonl          # Every resource touched
  ├── queries.jsonl         # Every query + result
  ├── index.json            # Hot index (reconstructible)
  └── cache/                # Short-term memory
      └── lineage-compose-py-251108.md

  ---
  Resource Types (Keep Simple)

  TIER 0 Resources:
  - file://local/path
  - https://url
  - git://repo-url
  - folder://local/directory

  Don't overclassify. Just log the URI.

  Access log captures:
  - What (URI)
  - When (timestamp)
  - Why (context/note from conversation)
  - Session (genealogy marker)

  (2) Three-Tier as Meta-Architecture

  The Profound Insight:

  Traditional: Data → Intelligence → Insights

  Three-Tier: Data (Tier 0)
                ↓
              Facts ABOUT data (Tier 1)
                ↓
              Usage ABOUT facts (Tier 2)
                ↓
              Meta-Intelligence (emerges from Tier 2)

  Example:

  Tier 0: https://graphiti.ai/docs (thing exists)

  Tier 1: {
    source: "https://graphiti.ai/docs",
    type: "documentation",
    authority: 1,
    tags: ["graph-memory", "temporal-graphs"]
  }

  Tier 2 (.nexus/access.jsonl): {
    accessed: [
      {session: "abc", context: "researching memory architectures"},
      {session: "def", context: "validating temporal approach"},
      {session: "ghi", context: "comparing to imem design"}
    ],
    git_validated: true,  # Code from graphiti appears in commits
    attention: 0.85
  }

  Meta-Intelligence (emergent):
  "Graphiti influenced imem's temporal approach, validated through 3 sessions, implemented in commits abc123,
  def456"

  Authority, interest, usage = Tier 2 metadata
  Intelligence emerges from patterns IN Tier 2

  (3) Query Results as Short-Term Memory

  The Workflow Shows:

  Lineage query took 30+ seconds, read 10+ files.

  The result is expensive knowledge. Cache it.

  .nexus/cache/
  ├── lineage-compose-py-251108.md      # Expensive synthesis
  ├── graphiti-influence-251108.md      # Cross-reference query
  └── trace-latest-251108.md            # Metadata query (cheap)

  Cache Properties:

  1. Time-bounded (24-48 hours? Or session-scoped?)
  2. Queryable (indexed in queries.jsonl)
  3. Reconstructible (not authoritative, can regenerate)
  4. Genealogy-linked (session_id ties cache to conversation)

  Cache Entry:
  {
    "query": "trace lineage compose.py",
    "result_path": ".nexus/cache/lineage-compose-py-251108.md",
    "session": "54a80...",
    "ts": 1699...,
    "ttl": "24h",
    "reconstruction_cost": "expensive",  # vs "cheap"
    "accessed_resources": [
      "git://commits/0807ef0",
      "file://.context/develop/.changes/251028-2300_flexgraph-compositional-memory.md",
      ...
    ]
  }

  Benefit:

  Ask tomorrow: "What's the lineage of compose.py?"
  → Check cache first
  → Serve cached if fresh
  → Regenerate if stale

  ~/.nexus/registry/                  # Or /knowledge/.registry/
  ├── graphiti-docs.md
  ├── anthropic-hooks.md
  └── react-three-fiber.md

  Tier 2 (Per-Project):
  PROJECT/.nexus/
  ├── access.jsonl                    # What this project touched
  ├── queries.jsonl                   # What this project queried
  └── index.json                      # This project's metadata index

  ---
  Why JSONL Not SQL

  1. Append-Only Matches Reality
  - Conversations are JSONL
  - Git is append-only DAG
  - Access logs are chronological
  - No updates, only additions

  2. Grep-able
  - grep "graphiti" .nexus/access.jsonl
  - No DB overhead
  - AI agents compose shell commands

  3. Reconstructible Index
  - index.json built from JSONL
  - Delete it anytime, rebuild from logs
  - No schema drift

  4. Portable
  - Copy .nexus folder = copy all metadata
  - No DB export/import
  - Works across machines

  5. Git-Trackable
  - JSONL diffs nicely
  - Can version metadata network
  - SQL dumps don't diff well

  What I Think

  MVP Storage:

  1. JSONL for logs (access, queries)
  2. JSON for index (reconstructible)
  3. Markdown for cache (synthesized results)
  4. No SQL/SQLite (wrong primitive for append-only)

  Three-Tier = Meta-Architecture:

  Yes. It separates:
  - Data (Tier 0)
  - Facts (Tier 1)
  - Usage (Tier 2)

  Intelligence emerges from Tier 2 patterns.

  Query Results = Short-Term Memory:

  Expensive syntheses should be cached with TTL.
  Cheap metadata queries can regenerate.

  The result is knowledge too:
  - Lineage report is knowledge
  - Store it, index it, make it queryable
  - Link it to session (genealogy)

  Non-destructive, accumulative:
  - Append-only logs preserve history
  - Index is reconstructible (not authoritative)
  - Cache is disposable (can regenerate)
  - Tier 1 stays inert (no project pollution)
  - Tier 2 accumulates (project-specific intelligence)

  Build it with JSONL.
