## The Innovation

**Traditional Vector DBs:**
- Chunk arbitrary text → embed → search semantically
- No schema, no structure, no guaranteed metadata
- Can't do: "Give me all Decisions where alternatives were considered"

**Traditional Structured DBs:**
- Define schema → insert records → query with SQL
- No semantic understanding, exact matches only
- Can't do: "Find decisions similar to this problem"

**IMEM:**
- Template enforces structure → LlamaIndex parses boundaries → Rich metadata + vectors
- Can do BOTH:
  - **Semantic:** "Find decisions about authentication" (vector similarity)
  - **Programmatic:** "Filter section_type='Decisions'" (metadata query)
  - **Combined:** "Find Decisions (filter) about auth (semantic) with alternatives considered (filter)"

## Why This Works

### 1. Template = Schema
```markdown
## Decisions
### Decision Title
- **Context**: ...
- **Solution**: ...
- **Alternatives**: ...
```

Every Decision "record" has consistent "columns" (Context, Solution, Alternatives).

### 2. LlamaIndex = Parser
Chunks at H3 boundaries → Each Decision = one chunk = one vector with metadata:
```python
{
    'section_type': 'Decisions',
    'section_name': 'Use JWT Authentication',
    'has_context': True,
    'has_solution': True,
    'has_alternatives': True,
    'content': 'Full section text'
}
```

### 3. Qdrant = Storage
Stores vectors + rich metadata → Enables hybrid queries

### 4. Python API = Query Engine
```python
# SQL-like precision
decisions = graph.search(
    query="authentication",
    filters={'section_type': 'Decisions', 'has_alternatives': True}
)

# Graph traversal
decision.constraints  # Related records in same "table"
decision.pattern_layer  # Foreign key to pattern "table"
```

## The Actual Innovation

**Using markdown templates + LlamaIndex as a schema enforcement layer for vector databases.**

**A structured vector database where:**
- **Schema** = Markdown template (H2/H3 structure + field definitions)
