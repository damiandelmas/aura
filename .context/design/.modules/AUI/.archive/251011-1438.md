  1. Brother Architecture: Persistent vs On-Demand?

  Option A: Persistent Watchers (Your Diagram)
  # 3 Claude instances running continuously
  SchemaValidator = claude -p --resume MAIN_SESSION --watch-mode schema
  LogicValidator = claude -p --resume MAIN_SESSION --watch-mode business_logic
  ArchValidator = claude -p --resume MAIN_SESSION --watch-mode architecture

  Pros:
  - Continuous intelligent validation
  - Each brother maintains domain expertise
  - Can catch subtle issues in real-time

  Cons:
  - 3x cost (3 Claudes running entire conversation)
  - --watch-mode doesn't exist in claude -p
  - Complex process management

  Option B: AUI Extracts → Spawn Brothers On-Demand
  # AUI watches JSONL (lightweight Python)
  for message in tail_jsonl():
      extractions = extract_dimensions(message)  # Regex/patterns

      if extraction.type == "schema":
          # Spawn brother JUST for validation
          validation = spawn_brother_validation("SchemaExpert", extraction)
          if user_confirms(validation):
              write_to_schema(validation)

  Pros:
  - Lower cost (spawn only when needed)
  - Simpler process management
  - AUI handles lightweight extraction

  Cons:
  - Brothers don't maintain continuous context
  - Less intelligent extraction (regex vs LLM)

  My Question: Do we need continuous intelligent watchers, or can AUI do lightweight extraction + spawn
  for validation?

  Flow:
  1. You say: "Payment can be declined"
  2. Tool use triggers PostToolUse hook
  3. AUI extracts: "Schema change detected: Payment.declined"
  4. AUI shows: "Add Payment.declined status? [y/n/details]"
  5. If [details]: Spawn SchemaExpert brother for validation
  6. Brother validates with you (separate prompt), writes to file
  7. AUI notifies main: "✅ Schema updated"
  8. Main reads file when needed

  Why this works:
  - Lower cost (on-demand spawning)
  - Simpler process management
  - Still gets intelligent validation when needed
  - File-based handoff (proven pattern)
