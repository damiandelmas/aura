### Phase 6: Relationship Primitives

**Goal:** Enable context assembly via validated primitives

**Effort:** ~150 lines, 1-1.5 days

**Validation:** Real usage showed 13+ needs for siblings operation

**Components:**

1. **siblings primitive** (+60 lines)
   - Get all chunks from same file_path
   - CLI: `imem siblings <result-id>`
   - Use case: Context assembly (decision → constraints → patterns)
   - Pain point: Validated 13+ occurrences in real session

2. **filter primitive** (+60 lines)
   - Session filtering: `--session <session-id>`
   - Temporal filtering: `--timestamp-before/after <time>`
   - Semantic filtering: `--semantic-similar <result-id>`
   - Use case: Genealogy traversal, temporal navigation

3. **combine primitive** (+30 lines)
   - Merge multiple result sets
   - Deduplication by result ID
   - Preserve metadata
   - Use case: Multi-query aggregation (pre-graph)

**Files:**
- `imem/src/imem/relationships.py` (new)
- `imem/src/imem/cli.py` (add subcommands)

**Usage pattern (Claude Code orchestrates):**
```bash

siblings = imem siblings <result-id>
conversation = imem filter --session <session-id>

# Claude assembles complete context response
```

**Validation:**

### Phase 7: Graph Operations

**Goal:** Enable authority ranking via graph algorithms

**Effort:** ~200 lines, 1.5 days

**Validation:** Real usage needed authority-based ranking

**Components:**

1. **graph build** (+100 lines)
   - Construct NetworkX graph from results
   - Add nodes (results) and edges (relationships)
   - Edges: file_path (siblings), session_id (genealogy), semantic similarity
   - Save to graphs/ directory
   - Return graph_id

2. **graph apply** (+80 lines)
   - Load graph from storage
   - Apply PageRank or betweenness centrality
   - Return reranked results

3. **graph export** (+20 lines)
   - Export to JSON (nodes/edges)
   - Optional: GEXF format for visualization

**Files:**
- `imem/src/imem/graph_ops.py` (new)
- `imem/src/imem/cli.py` (add graph subcommands)

**Validation:**
```bash
# Standalone usage
imem graph build <result-ids>
imem graph apply <graph-id> pagerank --top 10

# Or via batch (integrated)
imem batch '{"queries": [...], "graph": {"algorithm": "pagerank"}}'
```

**Usage in slash commands:**
```bash
# Multi-query authority ranking
r1 = imem develop search "auth" --decisions
r2 = imem develop search "auth" --failures
graph = imem graph build <r1+r2-ids>
ranked = imem graph apply <graph> pagerank
```

**Deliverables:**
- ✅ Query-time graph construction
- ✅ PageRank/centrality algorithms
- ✅ Authority-based ranking
- ✅ O(k²) not O(n²) (soft-graph property)

### Phase 9: Supersession Detection

**Goal:** Automatically detect when decisions are replaced

**Effort:** ~100 lines, 1-2 days

**Components:**

1. **Detection Algorithm** (+40 lines)
   - On new decision ingestion, search for similar existing decisions
   - Threshold: 0.85 semantic similarity
   - If found, mark older decision with `superseded_by` metadata

2. **Quality Impact** (+20 lines)
   - Update quality_score to heavily penalize superseded items
   - Rank superseded items lower in search results
   - Optionally hide superseded items (configurable)

3. **User Confirmation** (+40 lines)
   - Prompt user: "Does this replace [old decision]?"
   - Store supersession relationship if confirmed
   - Skip if user rejects

**Files:**
- `imem/src/imem/supersession.py` (new)
- `imem/src/imem/ingest.py` (integrate detection)

**Validation:**
```bash
# Index initial decision
imem init  # Indexes "Use JWT Auth" decision

# Index replacement decision
# Add new changelog: "Migrate to OAuth2"
imem init
# Prompt: "Does this replace 'Use JWT Auth'? [y/N]"
# If yes: Marks old decision as superseded

# Search shows newer decision ranked higher
imem search "authentication"
# Result 1: OAuth2 (current)
# Result 2: JWT (superseded, lower quality score)
```

**Deliverables:**
- ✅ Automatic supersession detection
- ✅ User confirmation workflow
- ✅ Quality score impact
- ✅ Optional hiding of superseded items

## Risk Assessment

### High Risk

**Template compliance dependency (CRITICAL - Foundation of entire system):**

**The moat:**
- Template-as-schema enforcement = unique differentiator
- MindsDB/Azure/e6data all do post-hoc extraction (probabilistic)
- AURA enforces at creation time (deterministic)
- Enables deterministic filtering: `filter(has_context=true, has_alternatives=true)`

**Current state:**
- Templates define required fields (Context/Solution/Rationale)
- Ingestion detects field presence (has_context, has_solution metadata)
- But doesn't reject violations (detection only, not enforcement)

**The risk:**
- All metadata guarantees depend on template compliance
- All filtering precision depends on deterministic fields
- All soft-graph relationships depend on reliable metadata
- If templates violated → system degrades to post-hoc extraction
- **Entire architecture collapses without template enforcement**

**Mitigation (Phase 8 - Non-negotiable):**
- Validation at ingestion (reject incomplete documents)
- Template compliance checking (required field verification)
- Schema evolution tooling (when templates change)
- This is not a feature — it's the foundational moat
- **Priority:** Must implement before widespread adoption

**LlamaIndex metadata parsing:**
- header_level not populated in v0.14.5
- Need to parse manually via regex
- **Mitigation:** Tested workaround exists (`ingest.py:695-697`)

### Medium Risk

**Soft-graph performance:**
- Metadata queries slower than indexed edges
- Multiple relationship traversals could be slow
- **Mitigation:** Session-scoped caching (Phase 6)

**Cross-collection search:**
- Qdrant doesn't natively support multi-collection queries
- Need custom aggregation logic
- **Mitigation:** Phase 11 marked low priority

### Low Risk

**Supersession false positives:**
- Semantic similarity threshold may trigger incorrectly
- User confirmation workflow mitigates
- **Mitigation:** Configurable threshold (default 0.85)
