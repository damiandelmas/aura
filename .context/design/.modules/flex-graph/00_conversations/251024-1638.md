  The Retrieval Paradigm Shift

  Traditional RAG:
  - Query → Get chunks → Hope they're relevant
  - No structure, no relationships, no genealogy

  What we have:
  - Query with dimensional filters (phase, layer, section, fields)
  - Returns self-contained knowledge units (H3 sections with Context/Solution/Rationale)
  - Provides navigation paths (same doc sections, pattern layer, conversation thread)

  The Options

  1. MARKDOWN RETRIEVAL (for AI agents)
  # Returns structured markdown
  result = search("database JSONB")

  ### Decision: Database as Inert Inventory Store

  - **Solution**: Migrated to metadata JSONB field...

  2. STRUCTURED JSON (for programmatic access)
  {
    "matched_section": {...},
    "related_sections": [...],
    "pattern_layer": {...},
    "conversation": {...}
  }

  3. GRAPH NAVIGATION (follow the genealogy)
  decision = search("database")
  → decision.constraints  # Related constraints
  → decision.pattern_layer  # Language-agnostic version
  → decision.conversation  # Ideation thread
  → decision.later_refinements  # Temporal evolution

  My Recommendation

  Build retrieval modes as separate CLI commands:

  # Quick lookup (current)
  imem develop search "database" --decisions

  # Full context
  imem develop explain "database JSONB"
  → Returns: Decision + Constraints + Implementation + Pattern

  # Decision archaeology  
  imem develop trace "database JSONB"
  → Returns: Conversation → Design docs → Final decision

  # Pattern learning
  imem develop patterns "provider agnostic"
  → Returns: Implementation + Pattern layer comparison

  Each mode returns markdown optimized for AI agent consumption.
