# Hybrid Knowledge Discovery Pattern

**Date:** 2025-10-25
**Status:** Pattern Specification
**Purpose:** Cross-project architectural pattern for flexible knowledge graphs

---

## Core Pattern

**Combine semantic matching with structural traversal to enable dynamic relationship discovery, where connections are established at query time rather than requiring predefinition at indexing.**

### The Hybrid Approach

```pseudocode
1. Semantic discovery: Find relevant concepts via similarity matching
2. Structural navigation: Access related items through metadata
3. Dynamic linking: Follow connections based on shared characteristics
4. Abstraction layering: Access generic versions of specific solutions
5. Temporal tracking: Discover evolution and supersession
```

**vs. Alternative Approaches:**
- **Pure semantic discovery:** Finds relevant items but loses relationships
- **Rigid predefined graphs:** Relationships fixed at design time, can't adapt
- **This pattern:** Discovers relationships dynamically from persistent metadata

---

## Relationship Discovery Categories

### 1. Structural (Grouped Items)
**Discovery mechanism:** Content source matching

```pseudocode
When item found:
  → Retrieve all related items from same source
  → May include constraints, solutions, patterns from same context
```

### 2. Abstraction Layer
**Discovery mechanism:** Naming convention pairing

```pseudocode
When accessing implementation item:
  → Check for language-agnostic counterpart
  → Format: implementation → abstraction variant
  → Enables cross-project reusability
```

### 3. Origin Tracing
**Discovery mechanism:** Session identifier matching

```pseudocode
When item found:
  → Trace back to originating discussion/ideation
  → Link to full context that created this item
```

### 4. Temporal Evolution
**Discovery mechanism:** Timestamp ordering + semantic matching

```pseudocode
When item found:
  → Search forward in time for refinements on same topic
  → Detect replacement/supersession relationships
  → Track changes over time
```

### 5. Cross-Context Pattern Matching
**Discovery mechanism:** Abstraction layer + semantic similarity across sources

```pseudocode
When pattern item found:
  → Query other contexts for equivalent patterns
  → Identify recurring solutions across domains
  → Aggregate similar approaches
```

## Example: Complete Item Genealogy

### Query: "How was performance issue resolved?"

**Phase 1: Semantic Matching**
```pseudocode
Search for: "performance issue"
Result found: Matching constraint (confidence score: 0.83)
```

**Phase 2: Structural Navigation**
```pseudocode
From constraint:
  → Parent item (created this issue): Found (earlier timestamp)
  → Context discussion: Link available (session identifier)

From parent item:
  → Resolution: Search forward for solutions on same topic
  → Resolution found: Item from later timestamp (2 weeks later)

From solution:
  → Pattern layer: Abstraction exists (language-agnostic version)
  → Pattern relationships: Found in 3 other contexts
```

**Output Hierarchy:**
```
Problem Found
├─ Caused by: Earlier decision
│  └─ Originating discussion: Session reference
├─ Resolved by: Later solution
│  └─ Generic pattern: Abstraction layer
└─ Similar patterns: Located in 3 other contexts
```

## Why Pure Alternatives Fail

### Semantic-Only Approach Cannot:
- Navigate from one item to causally related items without predefined links
- Discover connections between conceptually similar but differently-named items
- Handle dynamic relationship types (adapts as types evolve)

### Predefined Graph Approach Cannot:
- Find items by semantic meaning if not explicitly named
- Understand temporal causality and evolution
- Connect specific implementations to generic principles
- Trace origin discussions and decisions

### This Pattern Can:
- **Semantic discovery:** Find relevant items through meaning matching
- **Metadata navigation:** Access relationships through persistent attributes
- **Temporal navigation:** Trace evolution via timestamps
- **Abstraction navigation:** Link implementations to patterns via naming
- **Origin tracing:** Follow items back to their source discussions

## Discovery Metadata Structure

### Core Attributes for Each Item:
```pseudocode
{
    # Structural relationships
    'source_location': Unique identifier,
    'category': Type classification,
    'title': Human-readable name,

    # Temporal relationships
    'created_timestamp': ISO format datetime,

    # Origin relationships
    'origin_session': Unique session identifier,

    # Abstraction relationships (inferred)
    # Language-agnostic variant discoverable by naming pattern

    # Content for semantic discovery
    'content': Full text for embedding
}
```

### Relationship Discovery at Query Time:
1. **Grouped items:** Filter by source_location
2. **Abstraction variant:** Apply naming transformation
3. **Origin session:** Query by origin_session identifier
4. **Temporal evolution:** Filter by timestamp ordering + semantic similarity
5. **Cross-context patterns:** Query abstraction layer across all sources

## Dynamic Quality Scoring

### Calculated at Query Time:

```pseudocode
score = 1.0

Check if superseded by later item:
  → If yes, apply significant reduction

Count discovered constraints/issues:
  → Apply penalty proportional to count

Check if pattern abstraction exists:
  → Apply quality boost if present

Return combined score
```

### Ranking Application:
```pseudocode
For each search result:
  Calculate base score from semantics
  Calculate authority score from relationships
  Combine both scores
  Reorder results
```

## Pattern Serving Strategy

### When to Serve Generic vs Specific Version:

```pseudocode
If item is superseded and abstraction exists:
  → Return pattern (generic version)
  → Preserve reference to original
Else:
  → Return specific implementation
```

### Content Preserved in Generic Version:
- Context and motivation
- Solution principles
- Reasoning and analysis
- Discovered constraints
- Considered alternatives

### Content Abstracted Away:
- Technology-specific code
- Platform identifiers
- Framework names
- File paths and locations
- Stack-specific terminology

## Core Object Model

### Primary Classes:

```pseudocode
KnowledgeGraph:
  search(query) → returns matching items
  search_after(time, query) → returns items after timestamp
  trace_origin(session) → returns origin discussion

Item:
  # Lazy-loaded relationships
  related_items() → items from same source
  abstract_variant() → language-agnostic version
  origin_discussion() → session that created this
  later_evolution() → improvements/refinements
  quality_score() → computed authority
  is_current() → supersession status
  recommended_version() → best form to present

Pattern:
  source_implementation() → original specific version
  similar_across_contexts() → same pattern elsewhere

Discussion:
  resulting_items() → items created from this session
```

## Index Time vs Query Time

### Index Phase (Minimal):
- Parse source documents
- Extract metadata from structure
- Generate semantic embeddings
- Store with rich metadata
- Do not calculate relationships

### Query Phase (Intelligent):
- Semantic matching finds relevant items
- Instantiate domain objects
- Load relationships via metadata queries
- Compute authority scores
- Apply abstraction strategies
- Return ranked, contextualized results

**Key insight:** Relationships evolve as new items arrive. Cannot predict relationships at index time.

## Supersession Detection

### When New Item Added:

```pseudocode
1. Embed new item
2. Find semantically similar items
3. For each similar item above threshold:
   → Query user: "Does this replace existing item?"
   → If confirmed: Store supersession relationship
```

### When Serving Results:

```pseudocode
If item marked as superseded:
  → Check if abstraction exists
  → Serve abstraction if available
  → Otherwise note as outdated
Else:
  → Serve as current
```

## Multi-Context Pattern Accumulation

### Querying Generic Patterns Across Contexts:

```pseudocode
User has multiple contexts indexed

Query pattern across all contexts:
  1. For each context
  2. Search abstraction layer only
  3. Collect matches
  4. Return ranked by similarity
  5. Ensure no technology/language leakage
```

### Aggregation Logic:
```pseudocode
For each context:
  Search for pattern matches
  Filter to abstraction variants only
  Add to result set

Return all results ranked by relevance
```

## Design Decisions: Open Questions

### 1. Authority Score Caching
**Decision:** How long to keep computed scores?
- Session-temporary (recompute each query)
- Persistent (cache in metadata)
- Hybrid (cache frequently used)

### 2. Supersession Confidence Threshold
**Decision:** What similarity level triggers user prompt?
- Conservative (fewer false positives)
- Aggressive (catches more replacements)
- Configurable per context
- Default recommendation: 0.85

### 3. Temporal Authority Decay
**Decision:** Should older items lose authority over time?
- Linear degradation
- Exponential degradation
- Step function (cliff decline)
- Per-context settings

### 4. Pattern Extraction Recommendations
**Decision:** Should system suggest creating abstractions?
- Detect implementations without abstractions
- Flag generic concepts within specifics
- Prioritize by reusability potential

### 5. Multiple Embedding Models
**Decision:** Support different semantic models?
- Current standard model
- Domain-specific variants
- Code-specific embeddings
- Architecture: allows named vector variants

## Core Insight

**This pattern enables use cases impossible with pure semantic search or rigid predefined graphs by leveraging persistent metadata to discover relationships dynamically at query time, making relationships adapt as knowledge evolves without requiring design-time prediction.**
