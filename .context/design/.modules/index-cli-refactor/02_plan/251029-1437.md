## The Problem

**Current CLI (inconsistent):**
```bash

imem conversations search "query"  # noun-verb group
imem compose '{...}'           # verb-first
imem init                      # verb-first

```

**Issues:**
1. Inconsistent verb placement (groups use noun-verb, standalone uses verb-first)

4. Collection strategy undefined (per-project? global? mixed?)

## Decisions Made

### 1. Verb-Noun CLI Structure

**Rationale:** Natural English, consistent, extensible

**Two Clean Options

  Option 1: Flat (Hide .context/ implementation detail)

  imem index develop       # Implicit: .context/develop/
  imem index design        # Implicit: .context/design/
  imem index document      # Implicit: .context/document/
  imem index conversations # Different location (~/.claude/)

  Pros: Simple, short commands
  Cons: Doesn't show they're all "context"

  Option 2: Hierarchical (Show structure)

  imem index context         # All of .context/
  imem index context develop # Just .context/develop/
  imem index context design  # Just .context/design/
  imem index conversations   # Separate location

  Pros: Shows relationship
  Cons: Verbose, "context" in every command

  Hybrid (My Recommendation)

  # Common operations (short)
  imem index develop       # .context/develop/
  imem index design        # .context/design/
  imem index conversations # ~/.claude/

  # Convenience (index all phases at once)
  imem index context       # All of .context/ (develop + design + document)
  imem init                # Alias for 'index context'

  Why: Best of both - short commands for daily use, explicit context when you want
  everything.

  Search Follows Same Pattern

  imem search develop "query"
  imem search design "query"
  imem search context "query"      # Search all phases
  imem search conversations "query"

  Your Question: "Make context and design root level same thing?"

  I think you mean: Should design be a top-level source (like conversations), not nested
  under context?

  My take: Keep them separate in CLI:
  - design/develop/document = phases (under .context/)
  - conversations = separate source (under ~/.claude/)

  CLI treats them as peers (all top-level), implementation detail hidden.

**Recommended (Hybrid):**
```bash
imem index develop       # Phase (.context/develop/)
imem index design        # Phase (.context/design/)
imem index conversations # Source (~/.claude/)
imem index context       # All phases (convenience)
imem init                # Alias for 'index context'

imem search develop "query"
imem search design "query"
imem search conversations "query"
imem search context "query"      # All phases
```

**Why:** Short commands for daily use, explicit `context` when indexing everything.

### 2. Collection Architecture

**Decision: Separate collections by data type, same collection across phases**

```
Per-project:
  imem_{hash}_changelog     # All phases (develop/design/document)
  imem_{hash}_conversation  # Conversations only
```

**Rationale:**
- Changelogs (all phases) = structured decisions → Same vector space, filtered by phase metadata
- Conversations = exploratory dialogue → Different retrieval characteristics
- Separate collections prevent cross-contamination
- Phase metadata enables clean filtering within changelog collection

**Registry update needed:**
```python
# OLD
{'collection': 'imem_1ba1fff1'}

# NEW
{'collections': {
    'changelog': 'imem_1ba1fff1_changelog',
    'conversation': 'imem_1ba1fff1_conversation'
}}
```

### 3. BRAIN Architecture (Future, Don't Build Yet)

**Three-layer separation:**

```
Layer 1: Qdrant (immutable)
  - Chunks as written, never modified
  - {content, keywords, timestamp, ...}

Layer 2: BRAIN (~/.context/imem_brain.json, mutable)
  - {reference_count, superseded_by, pagerank_score, age_months}
  - Updated continuously (every query, graph op)

Layer 3: Entity Map (~/.context/imem_entities.json, evolves)
  - {"variant-system": ["variant-system", "variant system", "prompt variants"]}
  - Canonical term resolution
  - Updated periodically (weekly LLM prune agent)
```

**Query-time composition (ephemeral):**
```python
chunks = qdrant.search(query)           # Immutable
brain_state = brain.lookup(chunk_ids)   # Mutable metadata
graph = build_graph(chunks)             # Ephemeral
topology = detect_topology(graph)       # Ephemeral
enriched = enrich_context(chunks, graph, brain_state)  # Ephemeral
structured = structure_by_topology(enriched, topology) # Ephemeral
return render(structured, template)     # Ephemeral
```

**Don't build BRAIN yet** - needs usage accumulation (3-6 months). Phase 4 work.
