# IMEM: Flexible Knowledge Graph Specification

**Date:** 2025-10-25  
**Status:** Design Specification  
**Purpose:** Implementation guide for Python Graph API

---

## Core Concept

**IMEM combines semantic search with graph traversal to create a flexible knowledge graph where relationships are discovered at query time, not predefined at index time.**

### The Hybrid Approach

```python
# SEMANTIC: Find relevant nodes via vector similarity
results = graph.search("database performance")

# GRAPH: Navigate relationships from matched node
decision = results[0]
decision.constraints           # Same file relationship
decision.pattern_layer         # .pattern.md twin
decision.conversation          # session_id link
decision.later_refinements     # Temporal sequence
```

**vs. Traditional Systems:**
- **Traditional RAG:** Semantic search only, no relationships
- **Rigid KG:** Predefined edges, can't discover new connections
- **IMEM:** Semantic discovery + dynamic relationship traversal

---

## Relationship Types

### 1. Structural (Same File)
**Discovery:** `file_path` metadata match

```python
decision.constraints  # All constraints from same changelog
decision.failures     # All failures from same changelog
decision.patterns     # All patterns from same changelog
```

### 2. Abstraction Layer
**Discovery:** Filename pattern (`.md` → `.pattern.md`)

```python
decision.pattern_layer  # Language-agnostic version
# file: 20251011-1200_database.md
# → 20251011-1200_database.pattern.md
```

### 3. Conversational Origin
**Discovery:** `session_id` metadata match

```python
decision.conversation  # TRACE conversation that created this
# Links to full ideation thread
```

### 4. Temporal Evolution
**Discovery:** `timestamp` + semantic similarity

```python
decision.later_refinements  # Decisions after this timestamp on similar topic
decision.superseded_by      # If newer decision replaces this
```

### 5. Cross-Project Patterns
**Discovery:** Pattern layer + semantic similarity across collections

```python
pattern.similar_patterns  # Same pattern in other projects
# Returns pattern nodes from different codebases
```

## Example: Full Genealogy Traversal

### Query: "How did we solve slow queries?"

**Step 1: Semantic Retrieval**
```python
results = graph.search("slow queries")
# Returns: "JSONB Query Complexity" constraint (score: 0.83)
```

**Step 2: Graph Traversal**
```python
constraint = results[0]

# Who created this problem?
original_decision = constraint.parent_decision
# → "Database as Inert Inventory Store" (Oct 11)

# How did we ideate it?
conversation = original_decision.conversation
# → TRACE session: "JSONB vs tables" debate

# Did we fix it later?
solutions = constraint.solutions  # Temporal search forward
# → "Hybrid Storage Approach" (Oct 25, 2 weeks later)

# Is this a pattern?
pattern = solutions[0].pattern_layer
# → "Hot/Cold Data Separation" (language-agnostic)

# Has this pattern appeared elsewhere?
similar = pattern.similar_patterns
# → 3 other projects with same pattern
```

**Output:**
```
Problem: JSONB Query Complexity
├─ Caused by: Database as Inert Inventory Store (Oct 11)
│  └─ Discussion: "JSONB scales better" (session cb91d93d)
├─ Solved by: Hybrid Storage Approach (Oct 25)
│  └─ Pattern: Hot/Cold Data Separation (generic)
└─ Similar problems: 3 other projects
```

## Why This Is Impossible with Pure Systems

### Pure KG Can't:
- Find "slow queries" → "JSONB complexity" without predefined `causes` edge
- Discover semantic similarity between "performance issues" and "query complexity"
- Handle evolving relationships (what if we add new constraint types?)

### Pure RAG Can't:
- Navigate from constraint → causing decision → solution → pattern
- Understand temporal evolution of decisions
- Link implementation details to abstract patterns
- Connect conversations to resulting decisions

### IMEM Can:
- **Semantic discovery:** "slow queries" finds "JSONB Query Complexity" via embedding similarity
- **Structural navigation:** Traverse to parent decision via `file_path`
- **Temporal navigation:** Find later solutions via `timestamp` + semantic similarity
- **Abstraction navigation:** Jump to pattern layer via filename convention
- **Conversational navigation:** Follow to original discussion via `session_id`

## Metadata That Enables Relationships

### Stored in Qdrant Payload:
```python
{
    # Structural relationships
    'file_path': '20251011-1200_database.md',
    'section_type': 'Constraints',
    'section_name': 'JSONB Query Complexity',
    
    # Temporal relationships
    'timestamp': '2025-10-11T12:00:00-0700',
    
    # Conversational relationships
    'session_id': 'cb91d93d-2350-4287-bdf7-ec0a19b8ffc0',
    
    # Abstraction relationships (implicit)
    # .pattern.md twin discovered by filename
    
    # Content for semantic search
    'content': 'Full section text...'
}
```

### Relationships Discovered at Query Time:
1. **Same file:** Filter by `file_path`
2. **Pattern layer:** Replace `.md` → `.pattern.md`
3. **Conversation:** Query by `session_id`
4. **Temporal:** Filter by `timestamp >` + semantic similarity
5. **Cross-project:** Query patterns across all collections

## Runtime Authority Scoring

### Calculated Dynamically During Query:

```python
@property
def authority_score(self) -> float:
    base = 1.0
    
    # Check if superseded (temporal + semantic)
    later = self._graph.search_after(
        self.timestamp,
        query=self.section_name,
        limit=1
    )
    if later:
        base *= 0.3  # Heavy penalty
    
    # Check constraints discovered
    constraints = self.constraints  # Same file query
    base *= (0.8 ** len(constraints))
    
    # Check if has pattern (quality signal)
    if self.pattern_layer:
        base *= 1.2  # Boost
    
    return base
```

### Re-ranking at Query Time:
```python
results = graph.search("database approach")
for decision in results:
    decision._graph = graph  # Inject graph access
    score = decision.authority_score  # Triggers traversal
    combined_score = decision.semantic_score * decision.authority_score
```

## Pattern Swap Logic

### When to Serve Pattern vs Implementation:

```python
@property
def canonical_version(self):
    """Return pattern if superseded, else implementation"""
    
    # Check if superseded
    if not self.is_current and self.pattern_layer:
        pattern = self.pattern_layer
        pattern._superseded_from = self  # Track original
        return pattern
    
    return self  # Still current, show implementation
```

### What Gets Preserved in Pattern:
- ✅ Context (why decision arose)
- ✅ Solution principle (abstracted)
- ✅ Rationale (reasoning)
- ✅ Constraints (discovered blockers)
- ✅ Alternatives (options rejected)

### What Gets Discarded:
- ❌ Code snippets
- ❌ Framework-specific terms
- ❌ File paths
- ❌ Tech stack details

## Python API Design

### Core Classes:

```python
class KnowledgeGraph:
    def search(self, query: str, **filters) -> List[Decision]
    def search_after(self, timestamp, query, limit) -> List[Decision]
    def get_conversation(self, session_id: str) -> Conversation

class Decision:
    # Properties (lazy-loaded via graph traversal)
    @property
    def constraints(self) -> List[Constraint]
    @property
    def pattern_layer(self) -> Pattern
    @property
    def conversation(self) -> Conversation
    @property
    def later_refinements(self) -> List[Decision]
    @property
    def authority_score(self) -> float
    @property
    def is_current(self) -> bool
    @property
    def canonical_version(self) -> Union[Decision, Pattern]

class Pattern:
    @property
    def original_implementation(self) -> Decision
    @property
    def similar_patterns(self) -> List[Pattern]  # Cross-project

class Conversation:
    @property
    def resulting_changelogs(self) -> List[Decision]
```

## Query Time vs Index Time

### Index Time (Simple):
- Parse markdown with LlamaIndex
- Extract metadata from frontmatter
- Generate embeddings
- Store in Qdrant with rich payload
- **No relationship calculation**

### Query Time (Intelligent):
- Semantic search finds relevant nodes
- Instantiate Decision objects
- Lazy-load relationships via metadata queries
- Calculate authority scores
- Apply pattern swap logic
- Return ranked, contextualized results

**Why:** Relationships evolve. New decisions supersede old ones. Can't predict at index time.

## Supersession Detection

### At Changelog Creation:

```python
# New decision written
new_decision = create_decision(...)

# Embed and find similar
embedding = embed(new_decision.content)
similar = qdrant.search(embedding, limit=5)

# Check similarity threshold
for candidate in similar:
    if candidate.score > 0.85:
        prompt_user(f"Does this supersede {candidate.section_name}?")
        if confirmed:
            # Store supersession link
            new_decision.metadata['supersedes'] = candidate.id
            candidate.metadata['superseded_by'] = new_decision.id
```

### At Query Time:

```python
# Check if decision superseded
if decision.metadata.get('superseded_by'):
    # Serve pattern instead
    return decision.pattern_layer
else:
    return decision  # Still current
```

## Cross-Project Pattern Accumulation

### Query Patterns Across All Projects:

```python
# User has 3 projects indexed
# Each with isolated collection

# Query for authentication patterns
patterns = graph.search_patterns(
    query="authentication",
    projects="all"  # Query across collections
)

# Returns:
# - Project A: JWT pattern (Rust)
# - Project B: OAuth pattern (Python)
# - Project C: Session pattern (TypeScript)

# All language-agnostic
# No code contamination
```

### Implementation:
```python
def search_patterns(self, query: str, projects="all"):
    if projects == "all":
        collections = registry.list_all_collections()
    
    results = []
    for collection in collections:
        # Only search pattern layer
        hits = qdrant.search(
            collection=collection,
            query=embed(query),
            filter={'source': 'pattern'}  # Patterns only
        )
        results.extend(hits)
    
    return ranked_by_similarity(results)
```

## Open Design Questions

### 1. Caching Strategy
**Question:** Cache authority scores? Relationship traversals?

**Options:**
- Ephemeral (per-query session only)
- Persistent (store in Qdrant metadata)
- Hybrid (cache hot paths)

### 2. Supersession Threshold
**Question:** What similarity score triggers supersession prompt?

**Options:**
- 0.85 (catches most)
- 0.90 (fewer false positives)
- 0.95 (very conservative)
- User-configurable

### 3. Temporal Decay Function
**Question:** How should authority degrade over time?

**Options:**
- Linear: `1.0 - (age_days / 365)`
- Exponential: `0.95 ^ (age_days / 30)`
- Step function: No decay until 90 days, then 0.5x
- Project-specific rates

### 4. Pattern Extraction
**Question:** Should system suggest pattern extraction?

**Approach:**
- Detect implementation without pattern twin
- Flag if language-agnostic concepts present
- Prioritize by potential reusability

### 5. Multi-Model Support
**Question:** Support multiple embedding models?

**Current:** E5-Large-v2 (1024D)

**Future:** BGE, MiniLM, code-specific embeddings?

**Architecture:** Named vectors in Qdrant already supports this
