# The Vision: Knowledge Genealogy System

## What We're Building

**Not "better RAG"—a queryable knowledge genealogy system.**

AURA enables AI agents to navigate institutional memory through three dimensions (Time × Source × Structure) with guaranteed schema enforcement, semantic discovery, and graph-like traversal of decision lineages.

**Core insight:** Template-enforced markdown creates structured vector databases where every query has deterministic metadata, enabling SQL-like precision combined with semantic search and temporal genealogy navigation.

---

## The Three-Dimensional Knowledge Space

```
┌─────────────────────────────────────────────────────────┐
│                KNOWLEDGE DIMENSIONS                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  TIME ──────────────────────────────────►               │
│  (timestamps, evolution, supersession)                  │
│                                                         │
│  SOURCE                                                 │
│  ├─ Changelogs (WHAT WE BUILT)                        │
│  │   ├─ Phase: develop/design/designate/document      │
│  │   ├─ Layer: implementation/pattern                 │
│  │   └─ Sections: Decisions/Constraints/Failures      │
│  │                                                      │
│  └─ Conversations (HOW WE GOT THERE)                   │
│      └─ Links to changelogs via session_id             │
│                                                         │
│  STRUCTURE                                              │
│  └─ Fields: Context/Solution/Rationale/Alternatives    │
│      (Template-enforced, guaranteed present)           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Dimension 1: Time

**Temporal navigation through knowledge evolution:**

```python
# Find decision's evolution over time
decision = search("database schema design")[0]

# Navigate temporally
decision.earlier_explorations()  # Design phase discussions
decision.later_refinements()     # Subsequent improvements
decision.superseded_by()         # Newer replacement

# Time-based queries
search("auth", after_date="2025-10-01")  # Recent decisions only
```

**Enables:**
- Decision archaeology (trace genesis → evolution → current state)
- Supersession detection (identify outdated decisions)
- Temporal authority (newer solutions rank higher)

### Dimension 2: Source

**Knowledge comes from two tiers:**

**Tier 1: Changelogs** (validated outcomes)
- **Phase:** design (exploration) | designate (plans) | develop (validated) | document (stable)
- **Layer:** implementation (code-specific) | pattern (language-agnostic)
- **Sections:** Decisions | Constraints | Implementation | Failures | Options

**Tier 2: Conversations** (how we got there)
- User questions
- Assistant responses
- Code changes (patches)
- Tool usage
- Linked to changelogs via `session_id`

**Query examples:**
```bash
# Search validated knowledge
imem search "JWT auth" --in develop

# Search exploration
imem search "database options" --in design

# Search conversations
imem search "authentication" --in conversations --section "User Messages"
```

### Dimension 3: Structure

**Template-enforced fields create guaranteed metadata:**

```markdown
## Decisions
### Use JWT Authentication
- **Context**: Previous session-based auth caused scaling issues
- **Solution**: Migrate to stateless JWT tokens
- **Rationale**: Enables horizontal scaling without session store
- **Alternatives**: OAuth2 (too complex), API keys (less secure)
```

**Every Decision chunk has:**
- `has_context: true`
- `has_solution: true`
- `has_rationale: true`
- `has_alternatives: true`

**Enables deterministic queries:**
```python
# GUARANTEED to work (template enforced)
decisions_with_alternatives = search(
    "authentication",
    filters={
        'section_type': 'Decisions',
        'has_alternatives': True,  # This field ALWAYS exists
        'phase': 'develop'
    }
)
```

## Multi-Modal Retrieval Patterns

**Four retrieval modes = four graph traversal strategies:**

### Mode 1: SEARCH (Discovery)

**Purpose:** Find relevant knowledge semantically

```bash
imem search "database schema" --in develop --section Decisions
```

**Returns:** Section-level chunks ranked by similarity

**Use case:** "What did we decide about databases?"

**Graph pattern:** Semantic vector search with structural filters

---

### Mode 2: EXPLAIN (Context Retrieval)

**Purpose:** Get complete context for a decision

```bash
imem explain "database JSONB decision"
```

**Returns:**
- The decision (H3 section)
- Related constraints (same file)
- Implementation notes (same changelog)
- Pattern layer (abstraction variant)
- Origin conversation (session link)

**Use case:** "Why did we make this decision? What were the constraints?"

**Graph pattern:**
```
Decision (semantic match)
  ├─ Constraints (same file_path)
  ├─ Implementation (same file_path)
  ├─ Pattern Layer (.pattern.md variant)
  └─ Conversation (session_id match)
```

**Example output:**
```markdown
## Decision: Database as Inert Inventory Store
- **Context**: Original design embedded provider-specific columns...
- **Solution**: Migrated to metadata JSONB field...
- **Rationale**: Enables zero-SQL changes for new providers

### Related Constraints
- JSONB query complexity (filter on nested fields harder)
- Migration path from columnar to JSONB

### Pattern Layer
Generic principle: Prefer flexible schemas over rigid columns
when domain is rapidly evolving

### Origin Conversation
Session cb91d93d (2025-10-11, 45 minutes)
- User asked: "How to add new provider without schema changes?"
- Discussion explored: JSONB vs EAV vs polymorphic columns
- Outcome: JSONB chosen for flexibility/maintainability balance
```

---

### Mode 3: TRACE (Genealogy Navigation)

**Purpose:** Follow decision lineage from ideation to implementation

```bash
imem trace "database schema decision"
```

**Returns:** Chronological timeline
1. Initial conversation (ideation)
2. Design exploration (options considered)
3. Designate plan (chosen approach)
4. Develop changelog (implementation)
5. Later refinements (evolution)

**Use case:** "How did we arrive at this decision?"

**Graph pattern:**
```
Search semantic match
  ↓ (session_id)
Conversation (earliest timestamp)
  ↓ (session_id + phase=design)
Design exploration
  ↓ (session_id + phase=designate)
Plan document
  ↓ (session_id + phase=develop)
Implementation changelog
  ↓ (timestamp > self + semantic similarity)
Later refinements
```

**Example output:**
```markdown
## Decision Genealogy: Database Schema Design

### 1. Ideation (2025-10-11, Conversation cb91d93d)
User: "How to add providers without schema migration hell?"
- Discussed: JSONB, EAV, polymorphic columns
- Concern: Query complexity vs flexibility

### 2. Exploration (2025-10-12, Design Doc)
Evaluated three approaches:
- Option A: JSONB metadata field (flexible, complex queries)
- Option B: EAV pattern (ultra-flexible, poor performance)
- Option C: JSON column in Postgres (native support)

### 3. Decision (2025-10-13, Designate Plan)
Chose: JSONB metadata field
Rationale: Best balance of flexibility and performance

### 4. Implementation (2025-10-14, Develop Changelog)
Migrated provider data to metadata JSONB
- 12 provider-specific columns → 1 JSONB field
- Zero schema changes for new providers

### 5. Refinement (2025-10-18, Later Update)
Added GIN index on metadata field for faster queries
```

### Mode 4: PATTERNS (Abstraction Comparison)

**Purpose:** Find language-agnostic principles across contexts

```bash
imem patterns "provider agnostic design"
```

**Returns:**
- Implementation layer (code-specific)
- Pattern layer (abstraction)
- Similar patterns from other projects

**Use case:** "What's the reusable principle behind this implementation?"

**Graph pattern:**
```
Implementation chunk (semantic match)
  ↓ (naming: .md → .pattern.md)
Pattern abstraction
  ↓ (phase=pattern + semantic similarity across collections)
Similar patterns in other projects
```

**Example output:**
```markdown
## Implementation: Database Metadata JSONB
(Python/Django specific)
- Used Django JSONField
- PostgreSQL 14+ required
- Migration: 0023_provider_metadata.py

## Pattern: Flexible Schema via Structured Documents
(Language-agnostic principle)
- **Context**: Domain evolves rapidly, schema changes costly
- **Solution**: Store variant data as structured documents
- **Tradeoff**: Query complexity vs schema flexibility
- **Applicable**: Any domain with high variation
- **Avoid**: When relationships are stable and queryable

## Similar Patterns
1. MongoDB document model (NoSQL approach)
2. Elasticsearch dynamic mapping (search-optimized)
3. JSONB in Postgres (relational hybrid)
```

## Soft-Graph Navigation: The Object Model

**Relationships via metadata queries, not explicit edges:**

```python
from imem import KnowledgeGraph

graph = KnowledgeGraph(project_root="~/my-project")

# Semantic discovery
decision = graph.search("JWT authentication")[0]

# Lazy-loaded relationships (metadata queries)
decision.constraints          # Filter: same file_path + section_type='Constraints'
decision.pattern_layer        # Naming: decision.file_path → .pattern.md variant
decision.origin_discussion    # Filter: session_id match → conversation collection
decision.later_evolution      # Filter: timestamp > self + semantic similarity
decision.superseded_by        # Filter: is_superseded_by metadata

# Navigation chaining
auth_decision.constraints[0].origin_discussion.patches
# "Show me patches from the conversation that created this constraint"
```

### Why "Soft Graph"?

**Traditional graph DBs (Neo4j):**
- Explicit edges: `CREATE (d:Decision)-[:CAUSES]->(c:Constraint)`
- Predefined at write time
- Fast traversal (indexed edges)
- Rigid (can't discover new relationship types)

**AURA soft-graph:**
- Implicit edges: metadata queries
- Discovered at query time
- Flexible (new relationships emerge from metadata)
- Progressive (adapts as knowledge evolves)

**Trade-off:**
- Slower than indexed graph traversal
- More flexible than predefined schemas
- Ideal for: evolving knowledge, AI agent memory
- Not ideal for: high-frequency graph analytics

## Template-as-Schema Philosophy

**Human-readable schemas, version-controlled structure:**

### Why Markdown Templates?

**vs JSON Schemas:**
- ✅ Human-readable (developers write naturally)
- ✅ Git-friendly (diffs show structure changes)
- ✅ Self-documenting (template IS the documentation)
- ✅ AI-agent friendly (LLMs understand markdown)

**vs API Validation:**
- ✅ Creation-time enforcement (not runtime)
- ✅ Version-controlled (schema changes visible)
- ✅ No external tooling (just markdown files)

### Progressive Disclosure

**Complexity emerges from document structure:**

```markdown
# Simple changelog (3 H3 sections → 3 vectors)
## Decisions
### Use JWT Auth
- **Solution**: JWT tokens
### Use Redis Cache
- **Solution**: Redis for session cache
### Deploy to AWS
- **Solution**: ECS containers

# Complex changelog (15 H3 sections → 15 vectors)
## Decisions
### Authentication Architecture
- **Context**: (5 paragraphs)
- **Solution**: (3 options evaluated)
- **Rationale**: (detailed tradeoff analysis)
- **Alternatives**: (4 rejected approaches)
## Constraints
### (6 more H3 sections)
## Implementation
### (4 more H3 sections)
```

**Result:**
- Simple docs = fewer vectors (fast indexing)
- Complex docs = more vectors (surgical retrieval)
- No manual tuning required
- Structure drives chunking strategy

## Bidirectional Linking: Decision Genealogy

**Two-tier architecture with session-based links:**

```markdown
# Changelog (.context/develop/.changes/251018-1200_auth.md)
---
session_id: cb91d93d
---
## Decisions
### Use JWT Authentication
- **Context**: Session-based auth didn't scale...
```

```markdown
# Conversation (.claude/projects/my-project/conversations/cb91d93d.jsonl)
{"type": "user", "message": "How to handle auth for API?"}
{"type": "assistant", "message": "Consider JWT..."}
```

**Enables:**
```python
# Changelog → Conversation
decision = search("JWT auth", filters={'source': 'changelog'})[0]
conversation = decision.origin_discussion  # session_id match
print(conversation.user_questions)         # Original question

# Conversation → Changelog
conv = search("auth API", filters={'source': 'conversation'})[0]
changelog = conv.resulting_changelog       # has_changelog + changelog_path
print(changelog.decisions)                 # Final decisions
```

**Use cases:**
- "Show me the discussion that led to this decision"
- "Did this conversation result in a changelog?"
- "What decisions came from session XYZ?"

## Per-Project Institutional Memory

**Isolation via Qdrant collections:**

```bash
# Project A
cd ~/project-a
aura && imem init
# Creates collection: imem_<md5_hash_of_path_a>

# Project B
cd ~/project-b
aura && imem init
# Creates collection: imem_<md5_hash_of_path_b>

# No cross-contamination
```

**Registry tracks projects** (`~/.context/imem_registry.json`):
```json
{
  "/home/user/project-a": {
    "collection": "imem_a1b2c3d4",
    "indexed_at": "2025-10-25T12:00:00",
    "doc_count": 347
  },
  "/home/user/project-b": {
    "collection": "imem_e5f6g7h8",
    "indexed_at": "2025-10-25T13:00:00",
    "doc_count": 128
  }
}
```

**Benefits:**
- Independent knowledge bases per project
- No leakage between contexts
- Scales to unlimited projects
- Fast project switching

## Use Case: AI Agent Workflows

### Scenario 1: New Developer Onboarding

**Agent query:**
```python
# "Why do we use JWT for auth?"
result = imem.explain("JWT authentication decision")
```

**Agent receives:**
- Original decision + rationale
- Constraints discovered during implementation
- Conversation where it was discussed
- Pattern layer (reusable principle)

**Agent can now:**
- Understand context without asking humans
- See alternatives that were rejected (and why)
- Access full genealogy (exploration → decision → implementation)

### Scenario 2: Refactoring Safety

**Agent query:**
```python
# "Find all decisions related to database schema"
decisions = imem.search(
    "database schema",
    filters={'section_type': 'Decisions', 'phase': 'develop'}
)

# Check constraints
for decision in decisions:
    constraints = decision.constraints
    print(f"Decision: {decision.title}")
    print(f"Constraints: {[c.title for c in constraints]}")
```

**Agent receives:**
- All decisions about database
- Associated constraints for each
- Can validate refactoring doesn't violate constraints

### Scenario 3: Pattern Reuse

**Agent query:**
```python
# "How did we solve similar provider-agnostic design before?"
patterns = imem.patterns("provider agnostic")
```

**Agent receives:**
- Implementation (code-specific)
- Pattern abstraction (reusable principle)
- Similar solutions from other parts of codebase

**Agent can:**
- Apply proven patterns to new problems
- Avoid reinventing solutions
- Learn from past tradeoffs

## Design Principles

### 1. Progressive Disclosure
- Simple templates → simple chunking
- Complex templates → rich metadata
- Complexity emerges naturally from structure

### 2. Creation-Time Enforcement
- Schema defined before writing (templates)
- Metadata guaranteed at ingestion
- Deterministic query results

### 3. Git-Native Knowledge
- Markdown files (human-readable)
- Version-controlled templates
- Diffs show knowledge evolution

### 4. Lazy Relationship Discovery
- No predefined edges
- Relationships discovered at query time
- Adapts as knowledge evolves

### 5. Microservices Independence
- IMEM (vector search) operates independently
- TRACE (conversation parsing) operates independently
- Compose via file-based coordination

## What This Enables (Summary)

**Impossible in traditional vector DBs:**
- ❌ Can't guarantee metadata presence
- ❌ Can't navigate relationships
- ❌ Can't trace decision genealogy

**Impossible in traditional graph DBs:**
- ❌ Can't discover semantically similar nodes
- ❌ Can't adapt relationship types dynamically
- ❌ Can't use markdown as schema

**Possible in AURA:**
- ✅ Deterministic metadata (template enforcement)
- ✅ Semantic discovery (vector search)
- ✅ Graph navigation (soft edges)
- ✅ Temporal genealogy (time-based traversal)
- ✅ Multi-modal retrieval (search/explain/trace/patterns)
- ✅ Git-native version control (human-readable)

**Bottom line:** Knowledge genealogy system for AI agents—not better RAG, not graph DB, a new category.
