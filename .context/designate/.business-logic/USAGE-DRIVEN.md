# Usage-Driven Systems

**Core Insight: Systems should learn from behavior, not designer predictions.**

---

## What This Means

Observation:
- Track how agents actually use the system
- Count composition patterns
- Detect recurring behaviors
- No assumptions about what's valuable

Learning:
- Proven patterns (10-20 uses) become presets
- Preset library grows organically
- System self-improves from usage data
- No manual curation required

---

## Why

**Validation:** Usage reveals value, not designer hunches

**Discovery:** Agents find patterns designers didn't anticipate

**Organic growth:** Preset library emerges from proven need

**Self-improvement:** System gets better automatically

---

## Core Principles

**Start flexible** - No rigid patterns until usage proves value

**Observe behavior** - Track compositions, don't predict them

**Capture proven patterns** - 10-20 uses = pattern worth keeping

**Self-improving** - Preset library grows without manual intervention

---

## Anti-Patterns

When someone wants to design presets upfront, ask:

**Do we have usage data?**
- No data → speculation
- 20 uses → proven pattern

**Example:**
- ❌ Design 5 preset queries based on what we think agents need
- ✅ Observe 30 uses of genealogy+siblings composition → capture as preset

**Are we building or validating?**
- Build → primitives (enable flexibility)
- Validate → presets (capture proven patterns)

---

## Implications

Systems must support:
- Usage logging (track compositions)
- Pattern detection (which recur?)
- Automatic preset generation (threshold-based)
- Optional preset usage (shortcuts, not requirements)

Future sessions will:
- Use flexible primitives initially
- Discover presets for common patterns
- Benefit from collective usage patterns

They won't:
- Be forced into pre-designed workflows
- Miss optimizations from usage learning

Observe, then optimize.
