# FlexSchema Overview

## Definition

**FlexSchema** is a type resolution methodology that applies universal CORE dimensions at index time, then resolves chunks to domain-specific types at serve time using pre-defined templates and project context.

**The "Flex":** Same universal coordinates resolve flexibly to different domain types based on who queries, from where, and for what purpose.

---

## Step-by-Step Implementation

### One-Time Setup

#### Step 1: Define CORE Dimensions (Universal)

6 dimensions apply to ALL knowledge, ALL domains:

- **Interrogative:** WHO/WHAT/WHERE/WHEN/WHY/HOW
- **Valence:** GOOD/BAD/NEUTRAL
- **Abstraction:** CONCRETE/ABSTRACT/META
- **Epistemic:** KNOWN/HYPOTHETICAL/UNKNOWN
- **Temporal:** PAST/PRESENT/FUTURE
- **Structural:** ATOMIC/COMPOSITE/RELATIONAL

#### Step 2: Create Domain Template Registry

```yaml
# software_domain.yaml
Decision: {what: 0.8, why: 0.7, valence: good, epistemic: known}
Pattern: {why: 0.9, how: 0.85, abstraction: abstract}
Failure: {what: 0.75, why: 0.8, valence: bad, temporal: past}
```

```yaml
# legal_domain.yaml
Statute: {what: 0.9, valence: neutral, abstraction: abstract}
Precedent: {what: 0.85, why: 0.75, who: 0.7, temporal: past}
Argument: {why: 0.95, epistemic: hypothetical}
```

---

### Per-Chunk (Index Time)

#### Step 3: CORE Classification

```
Chunk: "We decided to use JWT because session state was complex"
        ↓
CORE Classifier (1 LLM call or small model)
        ↓
Coordinates: {
    what: 0.85,
    why: 0.75,
    valence: good,
    epistemic: known,
    temporal: present
}
        ↓
Store: chunk + vector + CORE coords
```

**Key:** Universal metadata applied immediately. Domain-agnostic. Same for all projects.

---

### Per-Query (Serve Time)

#### Step 4: Context Gathering

```
Query from: project-barbar
Domain: software_development
Context: authentication implementation
```

#### Step 5: Template Matching

```
CORE coords: {what: 0.85, why: 0.75, valence: good}
        ↓
Match against software_domain templates
        ↓
Closest match: Decision {what: 0.8, why: 0.7, valence: good}
        ↓
Distance: 0.08 (close match)
```

#### Step 6: Contextual Qualification

```
Domain type: "Decision"
Project context: barbar/auth (high attention)
Temporal validation: code still matches (drift=0)
        ↓
Qualification: "Ground truth Decision for barbar auth"
```

#### Step 7: Serve

```
Return: Chunk tagged as "Decision" with high confidence
        Qualified as "canonical" for this project
```

---

## The Flexibility

Same chunk, different query:

```
Query from: project-npta (compliance audit)
        ↓
Same CORE coords {what: 0.85, why: 0.75, valence: good}
        ↓
Match: Still "Decision" (universal structure)
        ↓
BUT qualification: "Reference example from similar domain"
        ↓
Serve: Same type, different authority/qualification
```

---

## The Power

1. **First document fully typed** (no corpus needed)
2. **Cross-domain transfer** (legal learns from software via shared CORE)
3. **Context-aware** (same chunk = different qualification per project)
4. **Flexible resolution** (CORE fixed, domain type contextual)

---

## Summary Formula

**FlexSchema = CORE (universal) + Templates (domain) + Context (project)**

- **Index:** Apply CORE (1x)
- **Serve:** Match template + Qualify by context (∞x)
- **Result:** Universal foundation, contextual interpretation
