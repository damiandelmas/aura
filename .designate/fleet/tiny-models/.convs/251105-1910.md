# Runtime Chunk Editing with Tiny Model

---

## The Task

**Input:**
- Original chunk text (512 tokens)
- CORE coordinates (6 values)
- Confidence score
- Context (relationship to other chunks in package)

**Output:**
- Edited chunk (~5% modification)
- Adjustments: hedge language, confidence markers, temporal qualifiers

---

## Is This Viable for Tiny Model?

**Yes. Perfect candidate.**

### Why It Works

**Single-turn:** One edit pass, done

**Bounded output:** 95% same, 5% edits (constrained generation)

**Structured input:** CORE values guide edits deterministically

**Clear success metric:** Does edit match confidence/temporal status?

---

## Edit Types

**Low confidence → Hedge:**
```
Original: "JWT is the best approach"
Edited: "JWT appears to be a suitable approach"
```

**Temporal drift → Qualify:**
```
Original: "Use Redis for caching"
Edited: "Used Redis for caching (note: implementation evolved)"
```

**High authority → Strengthen:**
```
Original: "We might use PostgreSQL"
Edited: "We use PostgreSQL (validated across 3 projects)"
```

---

## Architecture

```
Input encoding:
  - Chunk embeddings
  - CORE coords (6 values)
  - Confidence scalar
  - Temporal status
  - Relationship type

Tiny editor model (5-10M params):
  - Identify edit locations
  - Generate minimal modifications
  - Preserve 95% of original

Training:
  - Labeled examples of appropriate hedging
  - ~2,000-5,000 examples
  - Cost: ~$100-200
```

---

# Salient Insights & Architectural Decisions

---

## Core Insights

**1. Type Inference for Knowledge**
- Hindley-Milner principle applies beyond code
- Don't declare structure upfront—let it emerge from observation
- Universal foundations enable contextual resolution

**2. CORE as Universal Algebra**
- Six dimensions: interrogative, valence, abstraction, epistemic, temporal, structural
- Universal coordinates that resolve to domain-specific types at runtime
- Domain types = tensor products of CORE dimensions

**3. Authority Is Contextual, Not Intrinsic**
- Same chunk, different value per project
- Tier 2 wrappers qualify at serve time
- Purpose + intention determine meaning

**4. Foundation Models Create Structure, Tiny Models Exploit It**
- Foundation: Initial semantic structuring ($0.01/chunk, one-time)
- Tiny models: Classification, polish, tasks (free after $100-500 training)
- AI agents with tools: Intelligent composition on structured foundation

**5. The Technological Moment**
- 2025: Tiny specialized models now viable (nanochat: $100-1000 training)
- Recursive refinement works (TRM: 7M params beats massive models)
- Economic inflection: Train custom > Pay APIs forever

---

## Architectural Decisions

**Three-Tier Foundation**
```
Tier 0: Raw sources (heterogeneous)
Tier 1: Objective registry (normalized markdown)
Tier 2: Subjective gateways (per-project qualification)
BRAIN: Intelligence layer (orthogonal, operates across tiers)
```

**CORE Schema Implementation**
```
Index time: 1 LLM call → 6D coordinates (universal)
Store: Rich metadata enables programmatic queries
Query time: CORE + context → resolve domain type
Serve time: Qualify based on project attention
```

**Layered Intelligence Stack**
```
Layer 1: Tiny models add structure (CORE, domain, confidence)
Layer 2: Metadata creates queryable foundation
Layer 3: AI agents with tools compose higher-order structures
Layer 4: Runtime resolution based on context
```

**Economic Strategy**
```
Foundation models: Heavy lifting, one-time ($10-20/1000 chunks)
Tiny models: Repetitive tasks, trained once ($100-500)
Deterministic ops: Graph composition, filtering (free)
Break-even: ~2 months vs API-only approach
```

**Technology Philosophy**
```
Build on stable abstractions, not specific implementations
Pluggable components (swap models/backends freely)
Prove with APIs, productionize with tiny models
Ship fast, iterate based on real usage
```
