  The Three-Tier Memory System (Corrected Understanding)

  .design/ - Exploration & Intent

  .design/.changes/ - Design Journey (Chronological Blockchain)

  - AI agent captures insights during design exploration
  - Ground truth of THINKING (what were we considering? why?)
  - Chronological, immutable record of design decisions
  - Return here to understand "what were we exploring at that time?"

  Example:
  .design/.changes/251007-2300_agent-manager-research.md
  → "We explored AutoGen vs CrewAI for agent orchestration"

  .design/.modules/ - Implementation Staging (R&D Aggregation)

  - Aggregates R&D into sprints/PRDs/implementation plans
  - Staging for NEW things: "Adding LLM pipeline", "A/B Testing SDK"
  - Where experimental approaches wait before becoming real

  Example:
  .design/.modules/llm-pipeline/
  ├── 01_requirements.md
  ├── 02_architecture-options.md
  └── 03_implementation-plan.md

  ---
  .develop/ - Reality & Emergence

  .develop/.changes/ - Implementation Ground Truth (Validated Changelogs)

  - ACTUAL FACT of what happened in development
  - Created via slash command /log:develop (user-validated)
  - Emerge from USER↔AI conversation (user confirms accuracy)
  - Treated as immutable ground truth (what actually happened)
  - Drift over time: new changelogs supersede/extend previous work

  Example:
  .develop/.changes/250811-2157_requesty-api-simplification.md
  → FACT: "We simplified Requesty API from 8 methods to 3"
  → FACT: "Changed from .get().then() to await .fetch()"

  Critical: These are validated by user via slash command creation, therefore ground 
  truth.

  .develop/.modules/ - Feature Snapshots (Comprehensive Context)

  - Entire snapshots of a feature/module from implementation
  - For features requiring "warmer, more comprehensive understanding"
  - Staging area before integration into .document/
  - Eventually graduates and integrates into .document/architecture/ etc.

  Example:
  .develop/.modules/llm-pipeline/
  ├── architecture/
  │   ├── 01_pipeline-design.md
  │   ├── 02_model-selection.md
  │   ├── 03_prompt-engineering.md
  │   └── 04_evaluation-framework.md
  ├── code-samples/
  └── decisions/

  Flow: When LLM pipeline is mature → integrate into
  .document/architecture/llm-pipeline.md

  ---
  .document/ - Stable Truth (Maintained via Changelogs)

  Purpose: Static snapshot of codebase in semantic/geometric/geographic map terms

  Critical Insight: .document/ is ONLY ground truth INSOFAR AS it's maintained by
  .develop/.changes/!

  AI Agent Reading Pattern:
  1. Read .document/architecture/core-system.md     (static map)
  2. Read .develop/.changes/251007-*.md             (recent momentum)
  3. Synthesize: Static + Inertia = Current Reality

  Structure (Progressive Complexity):
  .document/
  ├── schemas/           # Ground truth parameters (immutable)
  ├── business-logic/    # Workflows and operations
  └── architecture/      # System structure and design

  Integration Pattern:
  .develop/.modules/llm-pipeline/architecture/*
      ↓ (matures & proves itself)
  Integrates into .document/architecture/llm-pipeline.md

  The Component Roles (Refined)

  1. TRACE - Conversation Archaeology

  Purpose: Extract context from Claude Code JSONL (what was discussed)
  Feeds: Agent system with conversation context
  Does NOT: Create changelogs (that's ChangelogAgent's job)

  2. PULSE - Intelligent Document Transformation

  Purpose: Read .develop/.changes/ → Update .document/ layers
  Intelligence:
  - Understands 3-layer taxonomy (schema/logic/arch)
  - Routes updates to correct document
  - Integrates (not appends) new knowledge
  - Handles .develop/.modules/ → .document/ graduation

  Critical: PULSE maintains .document/ based on .develop/.changes/ ground truth

  3. IMEM - Universal Search

  Purpose: Index all three tiers (design/develop/document)
  Enables:
  - "What were we thinking about agents in Sept?" → .design/.changes/
  - "What actually happened with the API?" → .develop/.changes/
  - "What's our current auth workflow?" → .document/business-logic/

  4. Agent System - Orchestrated Intelligence

  ChangelogAgent

  - Uses TRACE to extract conversation context
  - Creates .develop/.changes/YYMMDD-HHMM_topic.md
  - User validates via /log:develop slash command
  - Result: Ground truth changelog

  PulseAgent

  - Reads .develop/.changes/ (ground truth)
  - Updates .document/ layers appropriately
  - Handles drift (new changelog supersedes old work)
  - Integrates .develop/.modules/ when ready

  PruneAgent

  - Updates metadata on older .develop/.changes/ files
  - Marks which changelogs supersede others
  - Maintains temporal coherence in the blockchain

  ---
  Key Insights Synthesized

  1. Two Types of Ground Truth

  Design Ground Truth (.design/.changes/):
  - What we were THINKING
  - Blockchain of design decisions
  - Immutable historical record

  Development Ground Truth (.develop/.changes/):
  - What we actually DID
  - User-validated via slash commands
  - ACTUAL FACT (emerges from user+AI conversation)

  2. .document/ is Derivative

  .document/ is NOT ground truth by itself!

  It's only accurate when maintained by .develop/.changes/

  AI Reading Pattern:
  .document/          (static map - may be stale)
      +
  .develop/.changes/  (recent facts - momentum/inertia)
      =
  Current Reality

  3. Modules Graduate

  .design/.modules/feature/     (experimental, might fail)
      ↓ (if we build it)
  .develop/.modules/feature/    (proven, comprehensive snapshot)
      ↓ (when mature)
  .document/architecture/       (integrated into stable docs)

  4. Changelog Drift is Expected

  .develop/.changes/ files drift over time:
  - Newer changelogs supersede older ones
  - PruneAgent maintains metadata relationships
  - Search across both gives complete picture
